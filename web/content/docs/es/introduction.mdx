**InboxTap es un servidor local de captura de email y un SDK TypeScript para tests automatizados.**
Úsalo cuando un test de Playwright, Cypress, Vitest, Jest o Bun necesite recuperar un enlace de
verificación, magic link, enlace de restablecimiento, invitación, código de un solo uso u otro valor
de un email real generado por la aplicación bajo test.

InboxTap se ejecuta en localhost sin Docker, acepta el mensaje SMTP de la aplicación y expone el
resultado capturado al test mediante un SDK tipado o una API HTTP. No es un proveedor de email de
producción, buzón alojado, servicio de entregabilidad, almacén persistente ni visor manual de email.

**Datos clave**

- Los mensajes permanecen en la máquina local y nunca se reenvían fuera.
- Las direcciones generadas por el cliente aíslan tests paralelos sin registro en el servidor.
- El paquete se ejecuta con `npx` o `bunx` y es compatible con Node 20 y posteriores.
- El almacenamiento, el tamaño, los resultados y los tiempos de espera tienen límites explícitos.

## Por qué InboxTap

Los flujos de email se sitúan en una frontera incómoda de los tests end-to-end. Mockear el emisor
de correo omite la salida que recibiría tu usuario, mientras que un servicio de correo alojado
añade cuentas, latencia de red y buzones reales a un test que debería ser determinista.

InboxTap mantiene esa frontera en local. Acepta destinatarios arbitrarios, parsea el mensaje
entregado y expone el resultado al mismo test que disparó el email.

## Cómo funciona

1. Inicia InboxTap en local o junto a tu proceso de tests.
2. Apunta la aplicación bajo test a `localhost:1025`.
3. Dale a la aplicación una dirección única creada por `InboxTapClient`.
4. Espera el valor que necesitas a través del SDK o consulta la API HTTP local.

Los mensajes capturados incluyen el sobre SMTP, las cabeceras normalizadas, el texto y el HTML
decodificados, los enlaces HTTP(S) descubiertos, los códigos únicos de 4–8 dígitos y la fuente
RFC 822 en bruto.

## Seguridad y alcance

Por defecto, InboxTap enlaza SMTP y HTTP solo a las direcciones de loopback (`127.0.0.1` y `::1`)
y nunca reenvía un mensaje al exterior. Es para desarrollo y tests automatizados, no para la
entrega de correo en producción.

El servidor v<Version /> almacena los mensajes en memoria, retiene los 100 más recientes por
defecto, rechaza los mensajes de más de 5 MiB por defecto y limita las esperas HTTP de long-poll a
60 segundos. No ofrece persistencia, buzón en el navegador, flujo de adjuntos, webhooks,
autenticación SMTP ni STARTTLS.
